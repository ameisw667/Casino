## 🔑 3. The "Provably Fair" Protocol (Technical Deep-Dive)

Transparency is the foundation of trust. Every game outcome is verifiable.

### 3.1. Components
- **Server Seed**: A 64-character random string generated by the server.
- **Server Seed Hash**: SHA-256 hash of the server seed (provided to user *before* the bet).
- **Client Seed**: A string provided by the user (or randomized by the browser).
- **Nonce**: An incrementing counter representing the number of bets made with the current seed pair.

### 3.2. Outcome Generation (Pseudo-code)
```javascript
const combinedString = `${serverSeed}:${clientSeed}:${nonce}`;
const hash = hmac_sha256(combinedString, serverSeed);
const outcome = parseInt(hash.substring(0, 8), 16) % 10000; // Example for Dice
```

### 5.1. Game 1: "Crash" (High Frequency)
- **Concept**: A multiplier starts at 1.00x and grows exponentially until it "crashes".
- **Visuals**: A sleek SVG line graph with a "Vercel" glow effect.
- **Logic**:
  - Multiplier Growth: `1.06^seconds`.
  - Crash Point Formula: `E = 100, h = random_hex, e = 2^52; if (h % (E/5) == 0) return 1.00; else return ( (e*100 - h) / (e - h) ) / 100`.

### 5.2. Game 2: "Dice" (Interactive Slider)
- **Concept**: User bets if the roll will be over/under a target.
- **Visuals**: A glassmorphic slider with real-time probability updates.
- **Logic**:
  - Win Probability: `100 / (Target - 1)`.
  - House Edge: Fixed at 1% or 2%.

### 5.3. Game 3: "Roulette" (3D Wheel)
- **Concept**: Classic European Roulette.
- **Visuals**: 3D-modeled wheel using Spline or CSS 3D transforms.
- **Logic**: Handling multiple concurrent bets per user; instant payout calculation.

## 💰 6. Financial & Payment Infrastructure

### 6.1. Stripe Fiat Flow
- **Deposit**: Next.js Server Action creates a Stripe Checkout Session -> User pays -> Webhook credits wallet.
- **Withdrawal**: User requests withdrawal -> Admin approval (manual or automated based on risk score) -> Stripe Payout.

### 6.2. Atomic Balance Updates
To prevent "Race Conditions":
```sql
UPDATE wallets 
SET balance = balance - :betAmount 
WHERE user_id = :userId AND balance >= :betAmount;
-- Returns number of rows affected. If 0, the user had insufficient funds.
```

## 🛡 7. Security & Anti-Fraud Architecture

1.  **Balance Locking**: When a bet is placed, the amount is moved to `locked_balance` until the game completes.
2.  **Rate Limiting**: Implementation of `upstash/ratelimit` on all betting endpoints (e.g., max 5 bets per second).
3.  **Webhook Validation**: Strict signature verification for Stripe and Clerk webhooks.
4.  **Audit Logs**: Immutable logging of every balance change for forensic analysis.
5.  **DDoS Protection**: Vercel's built-in WAF + Cloudflare for the microservice.
